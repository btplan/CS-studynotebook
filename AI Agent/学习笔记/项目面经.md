# 自我介绍
这个项目是一个基于 LangGraph 和 LangChain 搭建的多 Agent 智能研究系统，主要解决复杂调研任务中“问题开放、信息分散、过程不可控”的问题。传统问答式 AI 往往只能一次性回答，很难把调研任务拆解清楚，所以我把整个流程设计成一个可编排的工作流：先由 Coordinator 做意图识别，再由 Planner 生成结构化研究计划，然后把任务分发给 Researcher 和 Coder 两类执行 Agent，最后由 Reporter 汇总成报告。

这个项目的难点主要在三个方面。第一是大模型输出不稳定，我通过 JSON 修复和 Plan 数据模型校验，把规划结果约束成结构化格式，保证计划可执行。第二是多 Agent 协作容易失控，容易出现状态混乱，所以我把状态、节点和跳转关系都显式建模，保证流程清晰。第三是 Agent 系统调试困难，我补了一套 trace 机制，把 LLM 调用、工具调用和节点执行过程都落到日志里，我还把模型、搜索工具、MCP 工具等能力做成可注入配置，方便测试和扩展

我认为这个项目的价值不只是做出了一个能跑的 Agent，而是把它工程化成了一个可扩展、可测试、可观测的研究框架。后续我还计划继续完善持久化状态、任务队列和上下文压缩，进一步提升长任务执行能力和生产可用性。

  

我这个项目是一个面向计算机考研 408 的本地知识库 RAG 问答系统。

项目背景是，408 包含数据结构、计算机组成原理、操作系统和计算机网络四门课，教材的标准和工业界前沿用的技术出入很大，普通大模型直接回答时容易出现表述不贴合教材、细节题幻觉。所以我做了一个基于本地资料的 RAG 系统，把教材、真题解析和题库资料构建成知识库，让模型先索依据，再生成答案。

针对 408 场景，我还设计了两点优化。第一是来源感知重排序，因为教材讲义、真题解析和普通题集的可信度不同，所以我在文档 metadata 中记录资料来源，并在重排序阶段加入来源权重，让系统优先采用更可靠、更贴合题型的资料。第二是题集专用切分策略，把一道题的题干、选项、答案、解析和知识点作为一个完整知识块，避免普通切分把题目结构拆散。项目还提供了 400 道选择题评测脚本，用来验证 RAG 系统在真实题库上的效果。

整体来说，这个项目不是简单调用大模型，而是围绕 408 垂直场景做了一个从资料处理、知识库构建、检索增强到题库评测的完整闭环。

### 3. 利用 Agent 和传统 LLM 应用的区别是什么？
答：传统 LLM 应用更像是固定步骤的输入输出映射，执行流程通常由开发者写死，模型本身不负责决策。

AI Agent 本质上是以 LLM 为决策核心的自主执行系统。模型不只是生成文本，还需要决定下一步做什么，比如调用工具、继续推理、请求人工确认，或者直接输出结果。

|方面|传统 LLM 应用|AI Agent|
|---|---|---|
|执行流程|开发者预定义，固定流程|动态流程，LLM 可自主决策|

|任务边界|单轮或固定多轮|开放式多步骤，可自我修正|

### 6. 为什么传统 logging 在 Agent 面前失效？
答：传统日志记录的是系统执行事件，而 Agent 系统除了执行过程，还有规划、推理、决策和工具协同等复杂行为。仅靠日志无法还原 Agent 的决策链路，因此需要 Trace、状态追踪和可观测性体系来观察 Agent 为什么这么做，而不仅仅是做了什么。

比如用户提出：“帮我分析英伟达未来三年的 AI 业务”。

如果只有传统日志，可能只能看到：

```text
Search Success
Search Success
LLM Success
```

但如果有 Trace，就可以看到完整链路：

```text
Planner
↓
生成 20 个步骤

Researcher
↓
第 7 步搜索失败

Planner Replan
↓
生成错误计划

Reporter
↓
引用空数据
```

### 7. 你如何对这个结果进行评测？
答：主要看完整率和信息搜索重复率。
	完整率：看每一个 step 是否有返回结果。
	重复率：每一个 search step 都会记录搜索 URL，最后计算重复 URL 的数量和比例。

### 8. Planner 拆出来的多个 steps，执行时是不是串行的？有没有像多子任务 fan-out / fan-in 那样并行跑？

答：每一个 step 是串行执行的。系统会选择第一个未完成的 step 分发给 Coder 或 Researcher，并且后续 step 的 prompt 会包含前面 completed steps 的 finding。所以整体是串行的 staged workflow，而不是 fan-out / fan-in 并行调度。

Coder 和 Researcher 内部是 ReAct Agent，会按照 ReAct 模式循环执行。主要并发发生在工具调用层面，比如 Agent 执行每一轮调用 LLM、调用工具、网络 I/O 时可以并发处理。

### 11. LangGraph 的协作方式有哪些？

答：

1. 顺序流水式：一个节点完成后进入下一个节点。
    
2. 条件路由式：根据 state 或模型输出决定下一跳。
    
3. Supervisor 模式：中心 Agent 统一调度多个 worker agent。
    
4. Handoff 模式：一个 Agent 完成后把控制权交给另一个 Agent。
    
5. Plan-and-Execute：先由 Planner 拆任务，再由执行 Agent 执行。
    
6. Fan-out / Fan-in：把多个子任务并发派发出去，再统一汇总。
    
7. Human-in-the-loop：中途插入人工确认、修改计划或审批。
    
8. Shared State / Blackboard：多个 Agent 通过共享 state 协作。
    

### 12. 你的多智能体是如何编排的？
答：我的多智能体系统基于 LangGraph。核心思想不是 Agent 群聊，而是状态机编排。
Planner 负责生成结构化计划，Researcher 和 Coder 分别执行检索与分析任务，Reporter 负责最终汇总。
所有 Agent 共享统一 State，通过 Graph 路由调度。Planner 输出会经过 Pydantic 校验，保证执行链路基于结构化数据运行。
本质上是 **LLM 负责推理，Graph 负责控制**。这样系统更稳定，也更容易观测和调试。
更具体来说，我的多智能体系统是基于 LangGraph 实现的，但本质上不是多个 Agent 自由聊天，而是一个有状态、有路由、有约束的工作流。
整体流程分为五个角色：
- Coordinator 负责意图识别。
- Planner 负责把用户需求拆解成结构化执行计划。
- Researcher 负责信息检索。
- Coder 负责代码分析和数据处理。
- Reporter 负责最终汇总结果并生成报告。
在编排层面，我采用状态机设计。所有 Agent 不直接通信，而是共享同一个 State。State 里面维护任务计划、执行结果和最终报告等信息。
路由由 LangGraph 控制，根据当前任务状态决定下一步执行哪个 Agent，而不是依赖 LLM 自己判断流程。
为了提高稳定性，Planner 输出的计划会经过 JSON Repair 和 Pydantic 校验，确保后续执行基于结构化数据，而不是自然语言。
当前采用串行执行策略，每次只执行一个步骤，结果写回 State 后再决定下一步动作。虽然吞吐量不是最高，但在 Research 场景下，可控性和一致性更好。
工程上还加入了工具注入、Trace 可观测性、人类反馈节点以及迭代次数限制等机制，避免系统变成一个无法调试的 Demo。

### 13. 如此编排的优势？

答：
1. 可控性强。自由协同最大的问题是流程不可预测，Agent 之间可能反复讨论、重复调用工具，甚至偏离用户目标。用显式编排以后，每个 Agent 的职责边界、输入、输出、跳转条件都是明确的，系统行为更稳定。
2. 结果一致性更好。多 Agent 自由对话时，最终结论很依赖上下文演化路径，同一个问题跑两次可能差异很大。我的方案里，Planner 先把任务结构化，再按步骤执行，最终输出围绕同一份 Plan 展开，一致性会高很多。
3. 工程上更容易调试和排障。自由协同一旦出错，很难判断问题出在谁的 prompt、谁的工具调用，还是上下文污染。显式编排下，问题可以定位到具体节点，比如是 Planner 计划质量差，还是 Researcher 检索不到有效结果，排查成本低很多。
4. 成本和延迟可控。多 Agent 自由协同通常会带来大量无效 token 消耗，因为它们会互相解释、互相确认。我的编排方式更像任务流，一次只让当前必要的 Agent 工作，能明显减少无效推理和工具调用。
5. 更适合接入业务约束。真实业务里经常要加规则，比如某一步必须先检索再分析、某类任务不能调用 Python、某些节点必须经过人工审核、任务最多执行几轮。这些约束放在自由协同里很难保证，但放在图编排里就很自然，因为路由本身就是规则。

所以我的判断是：
如果目标是探索式创意协作，开放式多 Agent 讨论有价值。
但如果目标是交付一个稳定、可测试、可观测的 AI 应用，我会优先选择显式编排。
我并不是否定自由协同，而是把它收敛在局部。宏观上用 Graph 控制流程，微观上允许单个 Agent 在自己的节点内部做 ReAct 推理，这样兼顾灵活性和稳定性。

### 15. 可靠性方面，如果继续往下做，你会怎么做？
答：第一，结构化输出更严格。Planner 现在依赖 JSON Repair，后续我会加强 schema validation、失败重试、格式校验，避免 plan 解析失败。
第二，工具调用加超时和重试。搜索、MCP、LLM API 都可能失败，所以要加 timeout、retry、fallback search engine。

第三，引用和事实校验。Researcher 输出的资料要保留 source，Reporter 生成报告时要检查引用是否真实、是否重复、是否冲突。

第四，结果一致性检查。多个 observation 之间可能矛盾，可以加一个 verifier / critic agent 做交叉验证。

第五，checkpoint 和恢复。LangGraph 支持 checkpointer，后续可以把中间状态持久化，失败后从某个节点恢复，而不是整条流程重跑。

第六，评测体系。建立一批固定问题，评估报告质量、引用准确率、工具成功率、耗时和成本。

第七，安全和配置治理。API key 不应该硬编码，工具权限也要限制，比如 Python REPL 默认关闭、沙箱执行。

### 20. RAG 文档分割的流程是什么？

答：RAG 文档分割一般是 **解析、清洗、切分、结构化、嵌入、入库、检索验证** 这一套流程。

1. PDF 解析。
    
    先判断 PDF 类型。如果是文本型 PDF，就用 PyMuPDF、pdfplumber、unstructured 这类工具提取正文、页码、标题、段落、表格和图片位置。
    
    如果是扫描版 PDF，就先走 OCR，把图片页转成可检索文本，同时保留页码和坐标。
    
2. 清洗和切分。
    
    去掉页眉页脚、页码、重复水印，然后按标题层级、段落语义、页码范围切 chunk，而不是固定每 500 token 硬切。
    
    每个 chunk 都要带 metadata，比如文档名、页码、章节标题、chunk 类型、坐标位置、是否来自表格或图片说明。
    
3. 向量化入库。
    
    普通文本 chunk 用 embedding model 转成向量，存到向量数据库。同时原文内容、页码、章节、表格 ID、图片 ID 也存到 metadata 里，方便回答时溯源。
    
4. 表格处理。
    
    如果 PDF 里有表格，不要只把表格转成普通文本。表格会单独抽取成结构化数据，比如 markdown 或 csv。
    
    然后做两份索引：
    
    - 一份是表格标题、表头、表注、摘要的向量索引。
        
    - 另一份是结构化表格本身，方便后续做精确查询、聚合和比较。
        
    
    比如用户问“哪个指标最高”，这种问题不适合靠 embedding 猜，要走表格结构化查询。
    
5. 图片处理。
    
    对于普通插图、流程图、曲线图、显微图，我会先抽取图片，保存图片文件和对应页码坐标；然后用多模态模型或 OCR / 图表解析模型生成 image caption，比如“第 3 页图 2 展示了模型整体流程”。
    
    这个 caption 会进入向量库。图片本体可以存对象存储，metadata 里保存 image_id、页码、位置和 caption。
    
    检索时先召回 caption，再把原图一起给多模态模型看，避免只靠文字描述丢信息。
    
    对于图表和曲线图，我会尽量抽取坐标轴、图例、曲线趋势和关键数值。如果是科研 PDF，我会特别保留 figure caption，因为 caption 通常信息密度很高。
    
6. 检索与生成。
    
    使用关键词检索 + 向量检索 + rerank。
    
    召回后根据 chunk 类型决定怎么处理：文本直接给 LLM，表格走结构化读取，图片用 caption + 原图，再交给多模态模型理解。
    
    回答时必须带引用，比如页码、表格编号、图编号，保证可追溯。
    

### 21. Trace 做了什么？

答：Trace 的目标不是直接评价答案好不好，而是先把系统运行过程中的原始事实记录下来。

因为在一个多 Agent 应用里，最终结果出问题，不一定是模型本身不行，也可能是 Planner 没规划好，Researcher 没搜到资料，Search Tool 没返回结果，或者 Reporter 没拿到前面整理出来的 observations。

所以我这次先做了一个最小可用的 Trace 系统，用来观察整条执行链路。

主要记录了四类信息：

1. Workflow 整体运行信息。
    
    也就是一次用户任务从开始到结束的完整 span，包括输入问题、开始时间、结束时间、运行状态，以及对应的 thread_id 和 trace_id。
    
2. 图节点执行信息。
    
    比如 coordinator_node、planner_node、researcher_node、reporter_node 这些节点，每个节点都会记录自己的 span_id、节点名称、开始时间、结束时间、状态和 attributes。这样我可以知道每一步是否执行成功、耗时多少。
    
3. LLM 调用信息。
    
    也就是每次模型调用的输入摘要、输出摘要、模型名称、token 使用情况、耗时和错误状态。这样如果一次任务变慢或者 token 消耗异常，我可以从 trace 里定位是哪一次模型调用造成的。
    
4. Search 工具调用信息。
    
    系统里 Researcher 会调用搜索工具，所以我也记录了搜索 query、搜索工具类型、返回结果里的 URL 等信息。这些数据后面可以用于评测，比如判断有没有足够的来源、来源是否重复、引用是否覆盖了最终答案。
    

实现上，我没有引入额外依赖，而是做了一个轻量级的 harness 模块。

每次运行时，系统会生成一个 thread_id 和 trace_id，然后把 trace 上下文通过 `RunnableConfig.configurable` 往下传递。这里我没有用全局变量，因为多 Agent 系统未来可能同时处理多个请求，如果用全局变量，不同请求之间容易串 trace。用 `RunnableConfig` 传递会更干净，也更符合 LangChain 和 LangGraph 的调用方式。

在图构建阶段，我把 callback handler 注入进去，并对节点做了一层包装。节点开始执行时创建 span，节点执行结束时关闭 span。如果中间发生异常，也会把错误状态写进 trace。

对于 LLM 和 Search 工具，我重点检查了调用链，确保 `llm.invoke`、`llm.stream`、`agent.ainvoke`、`search_tool.invoke` 这些地方都继续传入 config，这样 callback 链不会断，内部调用也能被记录下来。

### 22. 你怎么测评？baseline 是什么？

答：目前这个项目严格来说还没有做完整的标准化测评，只有两部分基础工作：

第一，做了节点级单元测试，验证 `coordinator / planner / researcher / coder / reporter` 这些核心节点能按预期工作。

第二，做了 Trace 采集，已经能记录 LLM 调用、工具调用、token 和执行链路，为后续评测打基础。

如果你问 baseline，我会坦诚说：当前仓库里还没有正式建立 baseline report。

但从评测设计上，我会把 baseline 定义成两类：

1. 单 Agent 直答。
    
    不做 `planner + researcher / coder` 编排，直接让一个 LLM 回答，这是最基础对照组。
    
2. 简化版 workflow。
    
    只保留 `coordinator -> planner -> reporter`，不做背景检索和分步执行，用来对比多智能体编排到底带来了多少增益。
    

后续我会重点评这几项指标：

- 任务成功率。
    
- 计划完成率。
    
- 重复搜索率。
    
- 平均延迟。
    
- token 消耗。
    
- 最终报告质量。

### 1. 如果给你一个课题，你会怎么完成？

答：如果给我一个课题，我不会一上来就直接写代码，而是先把它拆成一个可执行的闭环。

1. 问题定义。
    
    明确这个课题要解决什么问题、输入输出是什么、最终交付物是什么，以及评价标准是什么。
    
2. 资料调研。
    
    看相关论文、开源项目、已有方案和数据来源，判断这个课题是工程实现型、算法复现型，还是研究探索型。
    
3. 拆任务路径。
    
    把课题拆成数据、方法、实验、评估、报告几个模块，先设计一个最小可运行版本，保证能先跑通闭环。
    
4. 做实验迭代。
    
    先用 baseline 得到初始结果，再逐步优化模型、参数、数据和评估指标，同时记录每次实验的输入、输出和结论。
    
5. 做结果沉淀。
    
    把代码、数据处理流程、实验记录、失败原因和最终报告整理好，形成一个可复现、可解释、可交付的成果。
    

所以我的做法不是“想到哪做到哪”，而是先跑通最小闭环，再用实验结果不断迭代。最终交付的不只是结果，还有完整的方法路径和复现依据。
